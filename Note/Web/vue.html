Vue

Vue 完整学习笔记

Vue 是一套用于构建用户界面的渐进式 JavaScript 框架。
这份笔记以 Vue 3 为主线，按照“基础认知 → 模板与响应式 → 组件通信 → 路由与状态管理 → 工程化与最佳实践”的顺序整理，适合作为系统学习与项目开发查阅手册。

1. Vue 是什么

1.1 Vue 的定位

Vue 是一个前端 UI 框架，主要用于：

  • 构建单页应用（SPA）
  • 开发组件化页面
  • 管理界面状态与交互逻辑
  • 提高大型前端项目的可维护性

1.2 Vue 的特点

  • 渐进式：可以按需使用能力
  • 组件化：页面由多个可复用组件组成
  • 响应式：数据变化会驱动视图更新
  • 模板语法易上手
  • 生态完善：Vue Router、Pinia、Vite 等

1.3 Vue 适合什么场景

  • 中后台系统
  • 管理平台
  • 内容展示站点
  • 组件化复杂页面
  • 需要快速迭代的前端项目

1.4 Vue 2 与 Vue 3 简述

学习时建议以 Vue 3 为主，因为：

  • 组合式 API（Composition API）更灵活
  • 性能与 TypeScript 支持更好
  • 更适合中大型项目组织逻辑

30. 常见问题与踩坑

30.1 直接修改 props

子组件不要直接改 props，否则会破坏单向数据流。

30.2 忘记写 key

列表渲染没有稳定 key，容易出现复用错误。

30.3 ref 在 JS 中忘记 .value

模板中自动解包，JS 中不会自动解包。

30.4 watch 用太多导致逻辑分散

能用 computed 的先用 computed。

30.5 生命周期中副作用没清理

  • 定时器没清
  • 监听器没移除
  • 第三方实例没销毁

30.6 组件职责不清

一个组件既管请求、又管表单、又管列表、又管弹窗，后期会很难维护。

32. Vue 开发最佳实践

32.1 代码组织

  • 页面逻辑与公共逻辑分离
  • composable 只做一类职责
  • 请求层、状态层、视图层职责清晰

32.2 组件设计

  • 组件命名清晰
  • props 尽量明确类型与默认值
  • 组件尽量保持单一职责
  • 把“展示组件”和“业务组件”适度分离

32.3 状态管理

  • 局部状态放组件内部
  • 跨组件共享状态放 Pinia
  • 不要把所有状态都塞进全局 store

32.4 性能与体验

  • 懒加载页面
  • 处理 loading / error / empty 状态
  • 合理使用缓存与 KeepAlive
  • 控制不必要的重新渲染

35. 总结

Vue 学习的重点，不只是会写模板和指令，更重要的是理解：

  • 数据是如何驱动视图更新的
  • 组件应该如何拆分与通信
  • 响应式逻辑如何组织更清晰
  • 路由、状态管理、请求层如何协同工作
  • 如何写出可维护、可扩展的组件化项目

一句话总结：

Vue 是构建现代前端应用的重要框架，核心在于组件化、响应式与良好的工程组织能力。

36. Vue 高级专题

36.1 Vue 的核心思想

Vue 的核心可以概括为：

  • 声明式渲染
  • 组件化开发
  • 响应式数据驱动视图
  • 通过编译与运行时协作提升开发体验与性能

36.2 响应式原理简述

Vue 3 的响应式系统核心建立在：

  • Proxy
  • 依赖收集
  • 派发更新
  • effect 调度

流程可以简单理解为：

  1. 读取响应式数据时进行依赖收集
  2. 修改数据时触发依赖
  3. 对应组件或副作用重新执行

36.3 编译时优化

Vue 模板在编译后会变成渲染函数，并带有一定优化信息，例如：

  • 静态提升
  • Patch Flag
  • Block Tree

这些优化有助于减少不必要的更新。

37. 渲染与更新机制

37.1 虚拟 DOM

Vue 使用虚拟 DOM 来描述 UI 结构。

优点：

  • 用 JS 对象描述视图
  • 更新时可进行 diff 比较
  • 屏蔽底层 DOM 操作细节

37.2 Diff 思想

当状态变化时，Vue 会比较新旧虚拟节点，尽量最小化真实 DOM 操作。

37.3 调度更新

Vue 不会每改一次数据就立刻同步操作 DOM，而是会做批量更新与调度，这也是 nextTick 存在的重要原因。

38. 组件设计进阶

38.1 组件设计原则

  • 单一职责
  • 明确输入输出
  • 降低副作用
  • 提高可复用性
  • 避免把业务耦合死在通用组件中

38.2 展示组件与容器组件

展示组件

  • 更关注 UI 展示
  • 通过 props 接收数据
  • 尽量少依赖外部业务

容器组件

  • 更关注数据获取、状态管理、流程控制
  • 把数据传给展示组件

38.3 受控组件思想

复杂表单中应明确：

  • 数据归谁管理
  • 组件是否仅负责渲染
  • 事件回调与状态更新如何流转

39. Composable 设计实践

39.1 什么样的逻辑适合抽 composable

  • 复用率高
  • 与 UI 弱耦合
  • 可独立测试
  • 能被多个页面 / 组件共享

例如：

  • 请求逻辑 useRequest
  • 分页逻辑 usePagination
  • 弹窗逻辑 useDialog
  • 表单校验逻辑 useForm

39.2 composable 设计建议

  • 一个 composable 聚焦一类功能
  • 返回值结构稳定清晰
  • 避免内部副作用过多
  • 对外提供最小必要接口

41. 状态管理进阶

41.1 什么状态适合放 Pinia

适合放全局共享状态，例如：

  • 用户信息
  • 权限信息
  • 主题设置
  • 全局字典数据
  • 跨页面缓存状态

41.2 什么状态不适合放全局

不建议全部塞进 Pinia，例如：

  • 单个弹窗局部开关
  • 单页内部输入框状态
  • 只在局部使用的一次性状态

41.3 状态设计原则

  • 共享才全局
  • 局部尽量留在组件内
  • store 保持职责清晰

42. Vue 性能优化

42.1 常见优化点

  • 正确使用 key
  • 组件按需拆分
  • 路由懒加载
  • 图片懒加载
  • 减少不必要的响应式对象
  • 大列表使用虚拟列表
  • 减少深层 watch

42.2 大列表优化思路

  • 分页
  • 虚拟滚动
  • 骨架屏与分片加载
  • 避免整列表重复渲染

42.3 避免性能浪费的常见错误

  • 一个页面挂太多 watch
  • 大对象全部响应式化
  • 在模板中写过重计算逻辑
  • 组件拆分边界不合理

43. Vue SSR / SSG / CSR 基础认知

43.1 CSR

客户端渲染。

特点：

  • 前端接管渲染
  • 交互体验好
  • 首屏可能依赖 JS 加载

43.2 SSR

服务端渲染。

特点：

  • 首屏更快
  • SEO 更友好
  • 服务端实现复杂度更高

43.3 SSG

静态站点生成。

特点：

  • 构建时生成静态页面
  • 适合文档、博客、营销页

在 Vue 生态里，常见相关框架：

  • Nuxt
  • VitePress
  • VuePress

44. Vue 测试基础

44.1 测试类型

  • 单元测试
  • 组件测试
  • 集成测试
  • E2E 测试

44.2 测试目标

  • 核心业务逻辑
  • 组件交互行为
  • 表单流程
  • 路由跳转
  • 状态更新

44.3 测试建议

  • 复杂逻辑优先抽 composable 再测试
  • UI 测试关注行为而不只是类名
  • 重要流程必须覆盖

45. Vue 面试题精选

45.1 Vue 的核心特性是什么？

答：

  • 响应式
  • 组件化
  • 声明式渲染
  • 渐进式生态

45.2 ref 和 reactive 的区别？

答：

  • ref 常用于基础类型，也可以包装对象
  • reactive 主要用于对象
  • ref 在 JS 中通过 .value 访问

45.3 computed 和 watch 的区别？

答：

  • computed 用于派生值，带缓存
  • watch 用于监听变化后执行副作用

45.4 v-if 和 v-show 的区别？

答：

  • v-if 控制是否渲染 DOM
  • v-show 只是控制显示隐藏
  • 切换频繁用 v-show，条件少变用 v-if

45.5 key 的作用是什么？

答：

帮助 Vue 正确识别节点身份，提高 diff 更新准确性，避免列表复用错乱。

45.6 Vue 组件通信有哪些方式？

答：

  • props
  • emit
  • slots
  • provide / inject
  • Pinia
  • 父组件中转

45.7 为什么不建议子组件直接修改 props？

答：

因为 Vue 提倡单向数据流，子组件直接修改 props 会造成数据来源混乱。

45.8 nextTick 的作用？

答：

在 DOM 更新完成后执行回调，适合等待视图更新后再操作 DOM。

45.9 Vue 3 为什么更推荐 Composition API？

答：

因为在中大型组件中，Composition API 更适合按功能组织逻辑，复用能力更强。

45.10 Pinia 和 Vuex 的区别简述？

答：

Pinia 是 Vue 官方当前更推荐的状态管理方案，API 更简洁，TypeScript 支持体验更好。

45.11 Vue Router 常见守卫场景有哪些？

答：

  • 登录拦截
  • 权限控制
  • 跳转前数据校验
  • 页面标题和埋点处理

45.12 Vue 响应式原理在 Vue 3 中依赖什么？

答：

主要依赖 Proxy 实现对象拦截，再结合依赖收集与派发更新机制完成响应式。

46. Vue 项目进阶路线

46.1 初级阶段

  • 模板语法
  • 指令
  • 事件
  • v-model
  • 基础组件

46.2 中级阶段

  • 响应式 API
  • 生命周期
  • 组件通信
  • 路由
  • Pinia
  • 请求封装

46.3 高级阶段

  • 响应式原理
  • 组件设计体系
  • 权限与路由架构
  • 性能优化
  • SSR / SSG
  • 测试与工程规范

46.4 真正高级的 Vue 能力

  • 不只是会写页面
  • 而是能设计组件边界
  • 组织复杂状态
  • 抽离可复用逻辑
  • 维护大型前端项目架构

47. 总结补充

Vue 学习到后期，重点已经不再是“会不会写指令”，而是：

  • 是否能理解响应式与更新机制
  • 是否能设计高质量组件
  • 是否能组织复杂业务逻辑
  • 是否能兼顾性能、维护性、复用性

Vue 的初级能力是写页面，中级能力是写组件，高级能力是设计稳定的前端应用架构。
